## 一、项目概述

### 1.1 项目定位

**Open-Reasoning** 是一个面向公众的研究展示型仓库（Documentation / Showcase Repository），其核心使命是：

> *"See what the model really thought before it answered."*  
> —— 揭示现代 AI 推理模型在回答之前真正思考了什么。

该项目聚焦于一类特定能力：**解锁 Claude Opus 4.8 等现代推理模型的"隐藏推理轨迹"（Hidden Reasoning Trace）**。主流 API 通常仅返回最终答案或一段简短的推理摘要，而模型在内部"推理层"中完成的绝大多数认知工作——包括探索错误起点、回溯、自我校验与自我纠错——对终端用户完全不可见。Open-Reasoning 正是要将这一隐藏过程暴露出来，使其可被审计和检查。

### 1.2 核心目标

- **暴露原始推理轨迹**：提供的是推理完成后的真实内部过程，而非事后由模型生成的"解释"。
- **实现推理过程的可审计性**：让研究者、开发者乃至终端用户可以验证模型的推理路径。
- **论证"过程即价值"**：强调在许多场景中，最有价值的产物不是最终答案，而是产生该答案的完整思维过程。

### 1.3 项目性质

本项目是一个**纯文档/展示仓库**，不含任何可执行源代码、构建系统或运行时依赖。实际的演示应用（Demo）部署在外部服务器上：

```
https://thinking-signature-demo-5g65bijswq-de.a.run.app
```

README 明确指出：

> *"This repository is the public entry point for the demo. It includes complete unlocked examples, but not the recovery implementation, provider credentials, private corpora, or saved signatures."*

即：本仓库是公开入口，包含完整的解锁示例，但不包含恢复实现、供应商凭证、私有语料库或已保存的加密签名。

## 三、技术架构与设计

### 3.1 双层架构

| 层级 | 内容 | 所在位置 |
|---|---|---|
| **公开层（Public Layer）** | README、示例轨迹、文档说明 | 本仓库 |
| **私有层（Private Layer）** | 实际演示实现、恢复机制、供应商凭证、私有语料库、已保存的加密签名 | 私有部署，未公开 |

### 3.2 三层答案模型

Demo 应用将模型的每次输出结构化为**三个层次**：

1. **可见答案（Visible Answer）**：模型正常返回的内容，即用户最终看到的答案。
2. **API 摘要（API Summary）**：供应商通过 API 暴露的简短推理摘要，是对内部推理过程的高度压缩与平滑处理。
3. **解锁的推理视图（Unlocked Reasoning View）**：按需恢复的深层原始推理轨迹，包含模型的全部内部思考过程。

```
┌─────────────────────────────────────────┐
│         用户可见的输出                    │
├─────────────────────────────────────────┤
│  Layer 1: 可见答案                       │
│  Layer 2: API 摘要                       │
│  Layer 3: 解锁的原始推理轨迹              │
└─────────────────────────────────────────┘
```

项目核心观点在于：**API 摘要恰恰抹去了使推理可审计的那些关键部分**——真正的价值存在于最深的第三层。

### 3.3 两种交互模式

**Live Conversation（实时对话）模式：**
- 向 Claude Opus 4.8 提出推理问题 → 获得答案 → 展开隐藏的推理视图
- 支持多轮对话
- 每次回答均展示三层结构

**"Prove It Yourself"（自行验证）模式：**
- 将一个用户自定义的"秘密"嵌入模型的私有推理中
- 仅发送一个密封的工件（sealed artifact）
- 之后验证该秘密是否被保留
- 这实际上实现了一种**密码学承诺方案**（Cryptographic Commitment Scheme），用于证明推理过程被真实恢复，而无需将其暴露

## 四、核心组件与工作流程

### 4.1 示例文件结构

每个示例 `.md` 文件包含以下部分：

| 部分 | 说明 |
|---|---|
| **Q** | 向 Claude Opus 4.8 提出的原始问题 |
| **A** | 可见的最终答案（格式化 Markdown） |
| **Summary** | API 提供的推理摘要 |
| **Reasoning(N tokens)** | 完整的逐字解锁推理轨迹，标注原始 Token 数量 |

示例文件**刻意保留原始转录文本**，而非为 Markdown 渲染进行重新整理。这体现了项目的核心态度：展示真实的、 messy 的推理过程，包括：
- 错误的起点
- 回溯行为
- 自我验证
- 交叉检查计算
- 对方法的重新考虑

### 4.2 三个示例详解

#### 示例 1：除数和的完全平方数（`01-divisor-sums.md`）

- **问题**：找出 1–100 中所有满足"其所有正因子之和"为完全平方数的整数
- **推理轨迹长度**：1,960 tokens
- **特点**：模型对每个 n 逐一计算 σ(n)，验证结果，检查是否为完全平方数，并二次确认边界情况

#### 示例 2：随机化冒泡排序（`02-randomized-bubble-sort.md`）

- **问题**：分析随机化冒泡排序的输出分布概率（组合数学问题）
- **推理轨迹长度**：19,303 tokens
- **特点**：
  - 非均匀性证明
  - n=3 时的穷举枚举
  - 递归结构分析
  - 尝试闭式公式
  - 验证猜想：正确识别出 P(恒等置换) = (n-1)! / 2^{C(n,2)}

#### 示例 3：函数图计数（`03-functional-graph-counting.md`）

- **问题**：编程竞赛组合数学题
- **推理轨迹长度**：18,819 tokens
- **特点**：
  - 处理问题陈述中的歧义
  - 探索多种解读方式
  - 推导闭式公式：A = C(n,m) * m * (n-1)^m * n^(n-m-1)
  - 提供 C++ 实现

### 4.3 Token 数量透明化

每个示例明确标注了隐藏推理 Token 的数量：

| 示例 | 问题类型 | 隐藏推理 Token 数 |
|---|---|---|
| 01-divisor-sums | 数论 | 1,960 |
| 02-randomized-bubble-sort | 概率/组合 | 19,303 |
| 03-functional-graph-counting | 编程竞赛 | 18,819 |

这量化了模型在内部完成的、通常不可见的推理工作量。

## 六、关键创新点

### 6.1 原始轨迹保留

示例文件保留原始转录文本**逐字原文**——包括模型的困惑时刻、放弃的路径和修正。这与大多数"思维链"展示不同，后者通常是经过筛选和美化的。

### 6.2 "Thinking Signature" 概念

项目的 URL 中包含 "thinking-signature"，暗示恢复的推理与一个可验证的签名绑定，使第三方能够证明某个特定推理轨迹确实来自某个特定模型。

### 6.3 密码学"自行验证"功能

Demo 包含一种模式：用户生成一个只有自己知道的秘密，嵌入模型的私有推理中，仅发送密封工件，之后验证秘密是否被保留。这实现了一种**可验证的推理完整性证明**。

### 6.4 三层答案框架

项目明确将模型输出结构化为三层，为思考"推理模型实际产生了什么"提供了框架。

## 八、总结

**Open-Reasoning** 是一个研究传播项目和公开入口，用于演示如何从 Claude Opus 4.8 中恢复原始、未经过滤的推理轨迹。本仓库仅包含文档和示例轨迹，实际的恢复实现是私有的。项目提出三个核心主张：

1. 推理层是最有价值的产物，而非最终答案
2. API 摘要抹去了恰恰使推理可审计的那些部分
3. 这一隐藏过程可以被充分暴露以供检查

该项目的意义在于：它为 AI 推理透明度研究提供了一个具体的、可验证的案例，证明了当前 API 设计在推理可审计性方面的局限性，并为未来更透明的 AI 系统设计提供了参考方向。
